# v1.2.0 — Eliminación total de legado deploy-doc

**Fecha:** 2026-05-04
**Tipo:** Minor — breaking clean (sin cambios de API, solo rutas y nombres internos)

---

## 1. Solicitud

El usuario solicitó eliminar todas las referencias al binario y nombre `deploy-doc` del proyecto para que solo exista `gtt` de forma definitiva. Esto incluía:
- Binarios duplicados `deploy-doc-*` en los releases de GitHub
- Bloque de migración `isLegacy` en el updater (permitía actualizar desde un binario llamado `deploy-doc`)
- Directorio de instalación `Programs\deploy-doc` en Windows
- Path de configuración `~/.config/deploy-doc/config.yaml`
- Mensajes al usuario con referencias a `deploy-doc`

---

## 2. Motivación

La v1.1.1 renombró el CLI de `deploy-doc` a `gtt` pero mantuvo código de compatibilidad para usuarios del nombre anterior. En v1.2.0 se declara ese período de transición cerrado. Mantener el código legacy añade superficie de ataque, confusión en mensajes de error y archivos innecesarios en los releases.

---

## 3. Diseño técnico

### 3.1 Binario — updater.go
- `SelfUpdate(latest string)` pasó de retornar `(migrated bool, err error)` a solo `error`.
- Eliminado el bloque `isLegacy` que detectaba si el binario en ejecución se llamaba `deploy-doc` o `deploy-doc.exe` e instalaba `gtt` en su lugar.
- `CleanOldBinary()` simplificada: ya no busca ni elimina `deploy-doc.exe.old`.

### 3.2 cmd/update.go
- Actualizado para la nueva firma de `SelfUpdate`.
- Eliminado el bloque `if migrated { ... }` con el mensaje "Migración completada: deploy-doc → gtt".

### 3.3 Makefile — releases
- Eliminados los 3 `cp gtt-* → deploy-doc-*` de los targets `release`, `release-minor`, `release-major`.
- Eliminadas las 3 entradas `deploy-doc-linux/windows/darwin` de `gh release create`.
- Los releases de v1.2.0+ solo suben los binarios `gtt-*` y `checksums.txt`.

### 3.4 Instalador Windows — installer.go
- `InstallDir()` en Windows: `%LOCALAPPDATA%\Programs\deploy-doc` → `%LOCALAPPDATA%\Programs\gtt`.

### 3.5 Script PowerShell — scripts/install.ps1
- `$installDir` actualizado a `Programs\gtt`.
- Paso de limpieza [3/4] ahora elimina la carpeta `Programs\deploy-doc` completa (con `Remove-Item -Recurse`) en lugar de buscar archivos individuales dentro del directorio nuevo.

### 3.6 Migración de config — config.go + main.go
- `ConfigPath()` retorna ahora `~/.config/gtt/config.yaml`.
- Nueva función `MigrateIfNeeded()`: al arrancar v1.2.0+ por primera vez, si existe `~/.config/deploy-doc/config.yaml` y NO existe `~/.config/gtt/config.yaml`, copia el archivo al nuevo path y elimina el directorio viejo solo si la copia fue exitosa. Si la copia falla, el archivo original queda intacto.
- `main.go` llama `config.MigrateIfNeeded()` justo después de `updater.CleanOldBinary()`, antes de ejecutar cualquier comando.

### 3.7 Mensajes al usuario — todos los archivos
Actualizadas todas las referencias a `~/.config/deploy-doc/config.yaml` en mensajes de error y ayuda:
- `cmd/init.go`, `cmd/generate.go`, `cmd/qa.go`, `cmd/root.go`
- `internal/atlassian/jira.go`, `internal/atlassian/client.go`, `internal/atlassian/confluence.go`

## 5. Cómo verificar

```bash
# 1. Build limpio sin errores
go build ./...

# 2. Confirmar que no quedan referencias deploy-doc en mensajes al usuario
grep -rn "deploy-doc" --include="*.go" . | grep -v "github.com/geomark27/deploy-doc" | grep -v ".config/deploy-doc" | grep -v "MigrateIfNeeded copies"

# 3. Verificar la migración de config (simular usuario legacy)
mkdir -p ~/.config/deploy-doc
cp ~/.config/gtt/config.yaml ~/.config/deploy-doc/config.yaml 2>/dev/null || true
rm -f ~/.config/gtt/config.yaml
./bin/gtt version   # debe imprimir "✓ Configuración migrada a ~/.config/gtt/config.yaml"
ls ~/.config/gtt/config.yaml   # debe existir
ls ~/.config/deploy-doc/       # debe haber sido eliminado

# 4. Verificar que el Makefile no genera binarios deploy-doc-*
make build-all
ls bin/   # solo gtt-linux-amd64, gtt-windows-amd64.exe, gtt-darwin-amd64, checksums.txt
```

---

## 6. Notas de compatibilidad

- El config.yaml en sí no cambia de formato — solo cambia su ubicación en disco.
- Usuarios que tenían `~/.config/deploy-doc/config.yaml` migran automáticamente en el primer arranque sin intervención manual.
- No hay período de gracia: v1.2.0+ nunca escribe en `~/.config/deploy-doc/`.
