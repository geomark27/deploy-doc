# v1.1.0 — Minor (candidata a LTS)

> **Fecha:** 2026-04-03
> **Base:** v1.0.11 LTS

---

## Solicitud

Resolver limitaciones identificadas en v1.0.11 y crear estructura de documentación técnica en `docs/`:

1. La URL de Bitbucket en el builder usaba la org hardcodeada `devtyt` — se mantiene así intencionalmente ya que la herramienta es exclusiva para Torres y Torres.
2. `--flag=valor` (con `=`) fallaba silenciosamente en `parseFlags`.
3. El límite de búsqueda de docs previos era 5 — insuficiente en sprints avanzados.
4. `deploy-doc init` sobreescribía sin advertencia.
5. Solo se aceptaba un commit por servicio — algunos despliegues agrupan varios commits.
6. No había modo de previsualizar el ADF antes de publicar.
7. Crear documentación técnica en `docs/` como bitácora para futuros devs y agentes IA.

> **Nota:** el default de `Base URL` (`https://torresytorres.atlassian.net`) se mantiene intencionalmente — la herramienta fue creada para uso en esa organización.

---

## Motivación

Con la adopción del tool por el equipo completo y el uso en múltiples proyectos (que v1.0.11 ya soporta), las limitaciones técnicas acumuladas generaban fricciones. La falta de documentación técnica dificultaba que nuevos devs o agentes IA entendieran la arquitectura sin leer todo el código.

---

## Diseño técnico

### 1. Org de Bitbucket

La org `devtyt` se mantiene hardcodeada en `builder.go` igual que antes. La herramienta es exclusiva para Torres y Torres — no tiene sentido hacerla configurable y agregar complejidad innecesaria.

### 2. parseFlags con `--flag=value`

Se reescribió completamente en `cmd/generate.go`. El nuevo parser maneja 3 formas:
- `--flag valor` (espacio)
- `--flag=valor` (signo igual, con `strings.IndexByte`)
- `--flag` (booleano — se almacena como string vacía, presencia en mapa es lo importante)

Se mantuvo parser propio (no `flag.FlagSet`) por coherencia con el router manual y para no agregar complejidad por 3 flags.

### 3. Límite de búsqueda 5→10

Constante nombrada `deployDocSearchLimit = 10` en `confluence.go`. El valor 10 cubre sprints con muchos despliegues sin sobrecargar la respuesta.

### 4. Warning de sobreescritura en init

`runInit` verifica existencia del config con `os.Stat` antes de proceder. Si existe, muestra el path y pide `[s/N]` con default `N`. El bloque se insertó antes del banner de bienvenida para dar contexto antes de arrancar. El default de `Base URL` (`https://torresytorres.atlassian.net`) se mantiene — la herramienta fue creada para uso en esa organización.

### 5. Multi-commit (comma-separated)

Se agrega `GetChangedFilesMulti(hashes []string, workDir string)` en `git.go`. Itera sobre cada hash llamando `GetChangedFiles`, deduplica con `map[string]bool` preservando orden de primera aparición.

En `generate.go`, `splitHashes(raw string)` parte por coma y limpia espacios. Los hashes se pasan a `getFilesForCommits` (que llama `GetChangedFilesMulti`).

**Limitación conocida:** los links de Bitbucket en "Observación" usan el primer hash (`firstHash()`). Si un archivo fue modificado en el segundo commit, el link puede no apuntar al diff correcto. Aceptable para el caso de uso típico (un commit principal + hotfixes menores).

### 6. `--dry-run`

Flag booleano detectado con `_, dryRun := flags["--dry-run"]`. Si activo, imprime título + ADF JSON indentado a stdout (`json.NewEncoder + SetIndent`) y retorna sin contactar Confluence. No requiere credenciales válidas de Confluence.

### 7. BitbucketOrg en project add y configureProject

Se agrega prompt de `Organización en Bitbucket [devtyt]:` tanto en `runProjectAdd` (`project.go`) como en `configureProject` (`init.go`), usando `promptWithDefault` con default `"devtyt"`.

## Cómo verificar

```bash
# 1. --flag=value syntax
make run ARGS='generate --issue=APP-1 --commit-backend=abc123 --dry-run'
# Esperado: JSON ADF en stdout, sin contactar Confluence

# 2. Multi-commit + dry-run
# (parado en cualquier repo con commits reales)
make run ARGS='generate --issue APP-1 --commit-backend hash1,hash2 --dry-run'
# Esperado: archivos de ambos commits, sin duplicados, en el ADF

# 3. init — sin sobreescribir
make run ARGS='init'
# Si ya existe config → pregunta "¿Sobreescribir? [s/N]:"
# Con Enter o "n" → "Cancelado. Configuración sin cambios."

# 4. project add — ver prompt de BitbucketOrg
make run ARGS='project add'
# Debe aparecer: "Organización en Bitbucket [devtyt]:"

# 6. Búsqueda ampliada
make run ARGS='generate --issue APP-XXXX --commit-backend <hash>'
# Verificar que la lista muestra hasta 10 documentos previos (si existen)

# 7. Compilar con versión embebida
go build -ldflags "-X github.com/geomark27/deploy-doc/internal/build.Version=v1.1.0-test" -o /tmp/dd-test .
/tmp/dd-test version
# Esperado: deploy-doc v1.1.0-test
```
