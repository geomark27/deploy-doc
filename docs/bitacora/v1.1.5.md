# v1.1.5 — Security Patch

> **Fecha:** 2026-05-04
> **Base:** v1.1.4
> **Origen:** Reporte DevSecOps v1.1.4 — `docs/issues/desarrollo15.html`

---

## Solicitud

Infraestructura identificó 8 vulnerabilidades en el binario v1.1.4 mediante un escaneo DevSecOps (SAST + análisis de secretos + análisis de binario). Se solicitó remediar todos los hallazgos y documentar los patrones para evitar que se repitan en desarrollos futuros.

---

## Motivación

Las vulnerabilidades detectadas cubrían tres categorías:

1. **Información interna hardcodeada** — URLs corporativas, space keys de Confluence y el workspace de Bitbucket estaban literales en el código fuente. Un atacante con acceso al binario o al repositorio obtiene la topología interna sin esfuerzo. Los motores EDR/XDR (Trend Micro Vision One) detectan estas URLs como indicadores de C2.
2. **Backdoor de desarrollo** — `GTT_DEV=1` deshabilitaba la verificación de acceso al comando `gtt qa`. Clasificado como T1574 por los SIEM de la organización.
3. **Integridad del self-updater** — `gtt update` descargaba el binario sin verificar su hash, comportamiento idéntico al de un Dropper (T1105).

Además, la herramienta estaba acoplada a Torres & Torres (URLs, space key `"PA"`, workspace `"devtyt"`), lo que impedía usarla en otras instancias sin modificar el código fuente.

---

## Diseño técnico

### 1. Sistema de configuración por capas para space key y VCS

En lugar de simplemente mover los valores hardcodeados al YAML (lo que generaría fricción cuando un usuario necesita alternar entre espacios como `PA` y `ADN`), se implementó un sistema de prioridad de tres capas:

```
Flag CLI (--space ADN)           ← override puntual, sin editar archivos
  └── ProjectConfig.ConfluenceSpaceKey  ← default por proyecto (opcional)
        └── Config.ConfluenceSpaceKey   ← default global en config.yaml
```

Esto permite que:
- El usuario que siempre trabaja en `PA` configure su default una vez y no necesite flags.
- El usuario que alterna entre `PA` y `ADN` use `--space ADN` solo cuando lo necesita.
- La herramienta sea portable a cualquier instancia de Atlassian sin tocar el código.

El mismo patrón de dos capas aplica para `--vcs-host` y `--vcs-org`:
```
Flag CLI (--vcs-org mi-fork)     ← override puntual
  └── ProjectConfig.VCSHost/VCSOrg  ← configurado por proyecto en gtt init
```

### 2. Eliminación del backdoor GTT_DEV

Se eliminaron las 3 líneas del condicional en `cmd/qa.go`:

```go
// Antes — backdoor:
if os.Getenv("GTT_DEV") != "1" {
    if cfg.QAEmail == "" || !strings.EqualFold(...) { return error }
}

// Después — verificación siempre activa:
if cfg.QAEmail == "" || !strings.EqualFold(...) { return error }
```

El caso de uso legítimo que `GTT_DEV` resolvía (desarrollador probando el comando sin ser el usuario QA de producción) se cubre correctamente configurando `qa_email` con el email del dev en `config.yaml`.

### 3. Verificación de integridad en el self-updater

`SelfUpdate` ahora realiza dos requests en lugar de uno:

1. Descarga `checksums.txt` del mismo GitHub Release.
2. Descarga el binario.
3. Verifica SHA-256 del binario contra el hash en `checksums.txt` **antes de escribirlo a disco**.
4. Si el hash no coincide, aborta con error — el binario actual no es reemplazado.

El proceso `make release` fue actualizado para generar `checksums.txt` con `sha256sum` (Linux) o `shasum -a 256` (macOS) y subirlo junto con los binarios al GitHub Release.

Se eliminó también el `//nolint:noctx` reemplazando `http.Get(url)` por `http.NewRequestWithContext` con timeout de 120 segundos.

### 4. PATH hijacking en git

`git.GetChangedFiles` resuelve la ruta absoluta con `exec.LookPath("git")` antes de ejecutar. El LookPath redundante que existía en `cmd/generate.go:getFilesForCommits` fue eliminado — `git.go` es el único responsable.

### 5. URL de instancia Atlassian

`gtt init` ya no asigna silenciosamente `https://torresytorres.atlassian.net`. El wizard pregunta la URL al usuario y la persiste en `config.yaml` como `base_url`. El campo ya existía — solo faltaba pedirlo en lugar de hardcodearlo.

### 6. Migración para usuarios existentes

Al actualizar a v1.1.5, si `confluence_space_key` no está en `config.yaml`, `gtt g` y `gtt qa` muestran un error con las tres opciones de migración:

```
Opción 1 (recomendada): agrega confluence_space_key: PA a config.yaml
Opción 2: ejecuta 'gtt init' para reconfigurar todo
Opción 3: usa el flag en este comando: --space PA
```

### 7. Carga de config refactorizada

`config.Load()` siempre carga el archivo YAML y luego aplica env vars como override. El comportamiento anterior (solo cargar el archivo si algún env var estaba vacío) causaba que campos como `QAEmail` y `Projects` no se cargaran cuando los tres env vars principales estaban seteados. Este bug pre-existente quedó corregido como efecto colateral.

### 8. Bitácora de patrones seguros

Se creó `docs/security/patrones-seguros.md` — documento vivo con 7 patrones catalogados (P-001 a P-007), cada uno con anti-patrón, patrón correcto, clasificación OWASP/MITRE/PCI y una regla accionable para uso futuro en desarrollo manual o asistido por IA.

|---|---|---|
| `internal/config/config.go` | Modificado | Nuevos campos `ConfluenceSpaceKey` en `Config`, `ConfluenceSpaceKey` + `VCSHost` + `VCSOrg` en `ProjectConfig`. `Load()` siempre carga el archivo (fix carga parcial con env vars). Soporte para env var `CONFLUENCE_SPACE_KEY` |
| `internal/atlassian/confluence.go` | Modificado | `FindDeployDocByIssue`, `FindQAPage`, `FindQAPagesForModule` reciben `spaceKey string`. Si está vacío, buscan sin filtro de espacio |
| `internal/document/builder.go` | Modificado | `DeployDoc` tiene `VCSHost` y `VCSOrg`. `filesTable` los recibe como parámetro. Si están vacíos, las celdas usan texto plano en lugar de links |

| `internal/updater/updater.go` | Modificado | `SelfUpdate` descarga y verifica SHA-256 contra `checksums.txt`. Nueva función `fetchExpectedHash`. `http.Get` reemplazado por `http.NewRequestWithContext` |
| `cmd/init.go` | Modificado | Eliminada URL hardcodeada; wizard pide `base_url`, `confluence_space_key`. `configureProject` pide `vcs_host` y `vcs_org` |

| `cmd/root.go` | Modificado | Help rediseñado: sección de `gtt init`, prioridades de `--space` y VCS, ejemplos por escenario |

| `docs/security/patrones-seguros.md` | **Nuevo** | Bitácora de patrones seguros de desarrollo: P-001 a P-007 con evidencia, solución y regla accionable |

## Cómo verificar

```bash
# Build limpio
make lint
make build

# Verificar que gtt init ya no tiene URL hardcodeada
make run ARGS='init'
# Debe pedir: email, token, base URL, space key — en ese orden

# Verificar resolución de space key por capas
# (con confluence_space_key: PA en config.yaml)
make run ARGS='g -i APP-1 -b abc --dry-run'
# Usa PA del config

make run ARGS='g -i ADN-1 -b abc --dry-run --space ADN'
# Usa ADN del flag, ignora config

# Verificar que GTT_DEV ya no funciona
GTT_DEV=1 make run ARGS='qa -s 1 -m Test'
# Debe retornar "acceso denegado" si qa_email no coincide

# Verificar que exec.LookPath está activo
make run ARGS='g -i APP-1 -b abc --dry-run'
# Si git no está en PATH: "git no encontrado en PATH"

# Verificar help actualizado
make run ARGS='--help'
# Debe mostrar secciones: gtt init, flags de generate con prioridades, flags de qa, ejemplos

# Verificar checksums en release (solo al hacer release real)
make release
# Debe generar bin/checksums.txt con hashes de los 3 binarios
# GitHub Release debe incluir checksums.txt como asset
```
